## Lab 1.3: AWS AI-Powered Threat Detection — GuardDuty and Security Hub

This lab introduces two AWS-native AI/ML security services that require no agents, no scripts, and no custom configuration beyond enabling them. GuardDuty uses machine learning models trained on global threat intelligence to continuously analyze CloudTrail API calls, VPC flow logs, and DNS queries — detecting anomalies like port scanning, credential abuse, and unusual privilege activity. Security Hub aggregates findings from GuardDuty and other AWS services into a single prioritized view mapped to MITRE ATT&CK and compliance frameworks like CIS.

Together these tools automatically surface the suspicious activity you generated in labs 1.1 and 1.2.

> **Timing:** GuardDuty findings typically appear **10–30 minutes** after activity occurs. Security Hub findings backed by AWS Config (IAM, S3, EC2 posture) may take **30–60 minutes** on a freshly enabled account. Enable both services at the start of this lab, then complete Part 4 while you wait.

## Safety Rails

- Enable services only in your training AWS account.
- Do not apply IAM policy changes to production resources.
- Both GuardDuty and Security Hub offer a 30-day free trial — no charges during the lab.

## Part 1: Enable GuardDuty

GuardDuty is a managed threat detection service. Once enabled, it immediately begins analyzing data sources across your account using ML models — no agents, no log forwarding, no rules to write.

1. Open the AWS console and navigate to **GuardDuty**.

2. Click **Get Started**, then **Enable GuardDuty**.

![](/labs/images/guardduty-enable.png)

3. GuardDuty is now active. It will begin baselining your account's normal behavior and generating findings when anomalies are detected.

**What GuardDuty's ML models detect:**
- Reconnaissance: port scanning, unusual API enumeration
- Credential abuse: logins from unexpected locations, root credential use
- Privilege escalation: unusual IAM activity, EC2 metadata service abuse
- Command and control: communication with known malicious IPs or domains

---

## Part 2: Enable Security Hub

Security Hub is the aggregation layer. It collects findings from GuardDuty, AWS Inspector, Macie, and other services, then maps them to MITRE ATT&CK and industry compliance frameworks (CIS, PCI-DSS, NIST).

1. Navigate to **Security Hub** in the AWS console.

2. Click **Go to Security Hub**.

![](/labs/images/security-hub-get-started.png)

3. Leave the default standards selected — **CIS AWS Foundations Benchmark** and **AWS Foundational Security Best Practices** — and click **Enable Security Hub**.

4. Security Hub will begin ingesting findings. Leave this tab open and proceed to Part 4 while findings populate.

## Part 3: View Findings

> **If findings are not yet visible** — this is expected. GuardDuty takes 10–30 minutes to process recent activity. Complete Part 4 first and return here.

### GuardDuty Findings

1. Navigate to **GuardDuty → Findings**.

2. Look for findings related to the activity from lab 1.2:
   - `UnauthorizedAccess:EC2/SSHBruteForce` — from the failed SSH attempts

![](/labs/images/guard-duty-findings.png)

3. Click into a finding to see the full detail: affected resource, timestamps, MITRE ATT&CK technique, and GuardDuty's confidence score.

### Security Hub Findings

1. Navigate to **Security Hub → All findings**.

2. Filter by **Record state = ACTIVE** and sort by **Severity** descending.

![](/labs/images/security-hub-all-findings.png)

3. Findings include both GuardDuty-sourced threat detections and posture findings (IAM users without MFA, open security groups) from the CIS and FSBP standards.

4. You will likely see two types of identifiers in the findings:

   - **MITRE ATT&CK techniques** (e.g. `T1110 — Brute Force`) — describe *how* an attacker behaves. These appear on GuardDuty-sourced findings and tell you what the attacker was trying to do.
   - **CVEs** (Common Vulnerabilities and Exposures, e.g. `CVE-2024-12345`) — identify a specific, publicly documented software vulnerability with a unique ID. A CVE means a known flaw exists in a specific version of software that an attacker could exploit. These appear when Security Hub or Inspector scans your resources and finds unpatched software.

   The distinction matters: an ATT&CK technique tells you attacker intent; a CVE tells you a specific door is unlocked. A real incident often connects both — an attacker uses a technique (T1190 — Exploit Public-Facing Application) by exploiting a specific CVE.

## Part 4: S3 Misconfiguration — Create, Detect, Remediate

This part creates a real cloud misconfiguration, watches Security Hub detect it automatically, then remediates it. This is the full defender loop: introduce risk → detect → fix → verify.

### Step 1: Create a bucket with public access disabled

1. Navigate to **S3 → Create bucket**.
2. Name it something unique, e.g. `soc-lab-public-yourname`. Leave the region as `us-east-1`.
3. Under **Block Public Access settings**, **uncheck** "Block all public access" and confirm the warning.
4. Click **Create bucket**.

This simulates a common misconfiguration — a developer disabling the public access block to test something and forgetting to re-enable it.

### Step 2: Watch Security Hub detect it

Security Hub's **AWS Foundational Security Best Practices** standard includes control `S3.1` — it flags any bucket with public access block disabled.

1. Navigate to **Security Hub → Findings → All findings**.
2. Filter by **Title contains** `S3.1` or search for your bucket name.

> **Timing:** This finding may take **15–30 minutes** to appear. Continue to Step 3 and check back.

### Step 3: Remediate and verify

1. Navigate to your bucket in **S3**.
2. Go to the **Permissions** tab.
3. Under **Block public access**, click **Edit** and check **Block all public access**. Save.
4. Return to **Security Hub → Findings** — the finding will automatically update to **RESOLVED** within 30 minutes once AWS Config re-evaluates the bucket.

This closed loop — misconfiguration → automated detection → remediation → auto-resolution — is exactly how production security operations work.
