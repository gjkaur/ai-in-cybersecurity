
## Lab 1.1: Setting Up VM, SSH, and a Basic Service

In this lab, you will create an EC2 instance, connect to it from VS Code over SSH, practice Linux terminal and file-permission fundamentals, and run a basic web service that will be reused in later detection labs. You are building the telemetry infrastructure that AI models will analyze in later labs — without these logs and services, the automated investigations have nothing to work with.

## Safety Rails

- Run this lab only in your training AWS account.
- Do not run destructive commands against production systems.
- Keep all credentials (`.pem` key file) local to your machine — do not paste them into AI tools or share them.

## Attacker Objective (Cyber Lens)

Objective: Establish initial remote access to a cloud-hosted target.
ATT&CK focus: T1021 (Remote Services), T1078 (Valid Accounts).
Defender outcome: Validate that remote access is secured with key-only authentication and that baseline host and service visibility is in place before attack simulation begins.

## Learning Objectives

By the end of this lab, you will be able to:

1. Launch an EC2 instance and configure a security group for controlled SSH access.
2. Connect to a remote Linux host from VS Code over SSH using key-based authentication.
3. Navigate the Linux filesystem and apply file permission controls.
4. Identify privilege-escalation risk from SetUID binaries.
5. Run and validate a basic web service to generate workload telemetry for later labs.

## Prerequisites

- An AWS account with permissions to create EC2 instances, key pairs, and security groups.
- VS Code with the **Remote - SSH** extension installed.
- The `us-east-1` region selected in the AWS console (the AMI is region-specific).

> Be sure to use us-east-1, as the AMI is from that region.

1. Log in to AWS and open CloudShell. Then use the command below to create an SSH key.

```
aws ec2 create-key-pair \
  --key-name soc-lab-key \
  --query 'KeyMaterial' \
  --output text > soc-lab-key.pem
chmod 400 soc-lab-key.pem
```

> When pasting commands, be sure to hit "Enter" for last command 


2. From CloudShell, download the key before proceeding. 

![](/labs/images/aws-cloudshell-download.png)

3. Use CloudShell to create an EC2 instance. 
```
# create SG (firewall)
aws ec2 create-security-group \
  --group-name soc-lab-sg \
  --description "SOC Lab SG" || true

# get the default Virtual Private Cloud (network)
VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' \
  --output text)

# obtain the id of the SG recently created
SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=soc-lab-sg Name=vpc-id,Values=$VPC_ID \
  --query 'SecurityGroups[0].GroupId' \
  --output text)

# open SSH port 22 for SG
aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp \
  --port 22 \
  --cidr 0.0.0.0/0 || true

# open http port 80 for SG
aws ec2 authorize-security-group-ingress \
  --group-name soc-lab-sg \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0 || true

# Obtain default subnet
SUBNET_ID=$(aws ec2 describe-subnets \
  --filters Name=vpc-id,Values=$VPC_ID Name=default-for-az,Values=true \
  --query 'Subnets[0].SubnetId' \
  --output text)

# create ec2 instance
INSTANCE_ID=$(aws ec2 run-instances \
  --image-id ami-098e39bafa7e7303d \
  --instance-type t2.medium \
  --key-name soc-lab-key \
  --network-interfaces "AssociatePublicIpAddress=true,DeviceIndex=0,SubnetId=$SUBNET_ID,Groups=$SG_ID" \
  --query 'Instances[0].InstanceId' \
  --output text)
```

4. Obtain and store the public IP address using the instance ID. Wait 10-20 seconds for the public IP to become available.
```
aws ec2 describe-instances \
  --instance-ids $INSTANCE_ID \
  --query 'Reservations[0].Instances[0].PublicIpAddress' \
  --output text
```
> Note that these commands use variables. If your shell resets, variables will be lost. You can obtain IP address from the AWS console  under `ec2 -> instances -> public IP`. 

5. Open VS Code and navigate to the Remote Explorer extension (computer icon). Hover over SSH and click the settings icon.

![](/labs/images/remote-ssh.png)

6. When prompted, select the top option:

7. Replace the file with the contents below. Replace `<IP>` with your public IP  address from step 4. Update the IdentityFile path if needed. You can always get the soc-lab-key.pem path from the `file explorer -> Downloads -> right click file -> copy as path (or properties)`
```

  HostName <IP>
  IdentityFile "C:\Users\Administrator\Downloads\soc-lab-key.pem"
```

## Terminal Basics

1. Run through these basic commands to familiarize yourself with the Linux terminal.

```
ls
ls -l
cd /
ls -l
cd ~
pwd
mkdir -p test
cd test
touch test.py
cd ..              # only run once, this is relative path not absolute
ls -l test/test.py # make use of the tab key to auto-fill
```


2. Make a script executable and run it — this is how attackers and defenders both execute code on Linux.

```
# create file named hello.sh
echo 'echo Hello SOC' > hello.sh

# make script executable
chmod +x hello.sh

# execute script
./hello.sh
```

3. SetUID binaries run as their owner regardless of who executes them. Attackers look for these as privilege escalation paths.

```
ls -l /usr/bin/passwd
# notice the s — this binary is owned by root and runs as root for any user

# find all SetUID binaries on the system
find / -perm -4000 -exec ls -l {} \; 2>/dev/null
```

> **AI Assist:** Copy the output of the `find / -perm -4000` command above into any AI assistant (Claude, ChatGPT, Gemini, etc.) and ask: *"I am a security analyst. Which of these SetUID binaries are unusual or potentially risky on a standard Linux server, and why?"*

## Run a Basic Web Service (Required)

Running a tiny service gives you real workload telemetry for later labs.

1. Install and start nginx on the EC2 host.

```bash
sudo dnf install -y nginx
sudo systemctl enable --now nginx
sudo systemctl status nginx --no-pager
```

2. Replace the default page with a lab marker.

```bash
cat <<'EOF' | sudo tee /usr/share/nginx/html/index.html
<!doctype html>
<html>
  <head><title>SOC Lab Service</title></head>
  <body>
    <h1>SOC Lab Service Running</h1>
    <p>Training service endpoint active.</p>
  </body>
</html>
EOF
```

3. Validate service reachability from the instance and from your local machine/browser.

```bash
curl -I http://localhost
curl -s http://localhost | head -n 5
```

Using your EC2 public IP in a web browser:

```text
http://YOUR_PUBLIC_IP
```

4. Generate a few harmless requests so logs exist for later labs.

```bash
for i in {1..5}; do curl -s http://localhost >/dev/null; done
sudo tail -n 20 /var/log/nginx/access.log
```
