# Glossary: AI in Cybersecurity


---

## MITRE ATT&CK Framework

**ATT&CK** — Adversarial Tactics, Techniques, and Common Knowledge. A globally-accessible knowledge base of real-world attacker behaviors maintained by MITRE, used to classify and communicate threat activity.

**Tactic** — The adversary's high-level goal at a given phase (e.g., Initial Access, Privilege Escalation, Persistence).

**Technique** — A specific method used to achieve a tactic (e.g., Brute Force, Valid Accounts).

**TTP** — Tactics, Techniques, and Procedures. The full behavioral fingerprint of a threat actor or attack campaign.

**T1021 — Remote Services** — Using legitimate remote access protocols (SSH, RDP) to establish access to a target system.

**T1053 — Scheduled Task/Job** — Creating or modifying scheduled tasks or cron jobs to execute code persistently or at a specific time.

**T1059 — Command and Scripting Interpreter** — Using built-in scripting environments (bash, Python, PowerShell) to execute attacker-controlled code.

**T1078 — Valid Accounts** — Using legitimate credentials — stolen, guessed, or created — to authenticate to systems and blend in with normal activity.

**T1087 — Account Discovery** — Enumerating user accounts on a system or in a cloud environment to identify targets for further exploitation.

**T1003 — OS Credential Dumping** — Reading stored credentials from memory or disk (e.g., /etc/shadow on Linux, LSASS on Windows).

**T1110 — Brute Force** — Attempting many passwords or keys against an authentication mechanism to gain access.

**T1136 — Create Account** — Creating new local or cloud accounts to establish persistence on a compromised system.

**T1526 — Cloud Service Discovery** — Enumerating cloud services and configurations to identify misconfigurations, exposed resources, or attack paths.

**T1548 — Abuse Elevation Control Mechanism** — Bypassing or abusing `sudo`, `su`, SetUID binaries, or UAC to run code with higher privileges.

## SOC Operations

**SOC** — Security Operations Center. A team (or function) responsible for monitoring, detecting, analyzing, and responding to security events.

**Alert** — A notification generated by a security tool indicating a potentially malicious or anomalous event.

**Alert fatigue** — The desensitization of analysts caused by high volumes of low-fidelity alerts, which can cause real threats to be missed.

**Triage** — The process of quickly evaluating an alert or incident to determine its severity, credibility, and required response priority.

**Investigation** — A structured analyst workflow to determine the nature, scope, and impact of a security event.

**Hypothesis** — A candidate explanation for an anomalous finding, used in AI-assisted investigations to guide analyst evaluation.

**Incident** — A confirmed security event that requires response action. Distinct from an alert, which may be a false positive.

**Incident report** — A structured written record of an incident covering timeline, evidence, root cause, disposition, and recommended follow-up.

**Indicator of Compromise (IOC)** — A forensic artifact (IP address, hash, domain, log pattern) that indicates a system has been compromised.

**Indicator of Attack (IOA)** — Behavioral evidence of an attack in progress, regardless of the specific tool or malware used. IOAs focus on what the attacker is doing, not what they used.

**False positive** — An alert that triggers on benign activity incorrectly identified as malicious.

**False negative** — A missed detection — malicious activity that did not trigger an alert.

**Risk score** — A numeric value assigned to a security event to represent its priority for analyst review, typically based on severity, confidence, and context.

**Threat hunting** — Proactive, hypothesis-driven search through data to find threats that automated detection has not flagged.

**Kill chain** — A model of the sequential phases of a cyberattack: reconnaissance → delivery → exploitation → installation → command & control → exfiltration.

**Telemetry** — Raw event data collected from endpoints, networks, or cloud services and shipped to a monitoring system.

## Authentication and Identity

**SSH (Secure Shell)** — A cryptographic protocol for secure remote access to Linux systems. Used throughout the labs to connect to EC2 instances.

**SSH key pair** — An asymmetric cryptographic key pair (public + private) used for passwordless SSH authentication. The `.pem` file is the private key.

**Key-only authentication** — SSH configuration that disallows password-based login, requiring a private key for access.

**Brute force** — Automated repeated login attempts using a list of passwords or usernames.

**Credential stuffing** — A type of brute force that uses previously leaked username/password pairs to try to log in to other services.

**MFA (Multi-Factor Authentication)** — Requiring a second verification factor beyond a password. Root accounts and all IAM users should have MFA enabled.

**PAM (Pluggable Authentication Modules)** — A Linux framework that provides authentication services to applications. SSH, sudo, and `su` all go through PAM; it writes events to auditd.

**sudo** — A Linux command that allows a permitted user to run commands as root or another user. A primary target for privilege escalation.

**su** — Substitute User. A command to switch to another user account. `su - root` attempts to get a root shell.

**sudoers** — The configuration file (`/etc/sudoers` and `/etc/sudoers.d/`) that controls which users can run which commands with sudo.

**NOPASSWD** — A sudoers directive allowing a user to run sudo commands without entering a password. A high-risk configuration when applied broadly.

**Valid account abuse** — Using a legitimate account (owned or stolen) to perform unauthorized actions, blending in with normal user activity.

**Principle of least privilege** — Granting each user, process, or system only the minimum permissions needed to perform its function.

## Linux Endpoint Security

**auditd** — The Linux Audit Daemon. Monitors and records system calls and events defined by audit rules. Writes to `/var/log/audit/audit.log`.

**auditctl** — The command-line tool for managing auditd rules at runtime. Used to watch files, syscalls, and system events.

**ausearch** — A tool for searching auditd logs by key, user, or event type.

**audit.log** — The raw log file written by auditd, located at `/var/log/audit/audit.log`.

**Audit rule** — A directive loaded into auditd that specifies which system events to log. Example: watch `/etc/shadow` for reads.

**Audit key (`-k`)** — A label attached to an auditd rule used to filter events with `ausearch -k`.

**Auditd event type** — Structured type fields in audit log events:
- `USER_AUTH` — authentication attempt
- `USER_ERR` — failed authentication due to invalid user
- `USER_CMD` — sudo command executed
- `USER_LOGIN` — login event
- `USER_START` — PAM session start
- `ADD_USER` — new user created

**auid** — Audit UID. The original user ID at login time, preserved even if the user switches accounts with sudo. Useful for attributing activity to the originating person.

**res=failed** — Auditd result field indicating an operation was denied or failed.

**journald** — The systemd journal logging system. Aggregates logs from services, the kernel, and PAM.

**journalctl** — The command-line tool for querying journald logs. Supports filtering by service (`-u sshd`), user (`_COMM=sudo`), and time range.

**SetUID** — A Linux file permission bit (`s`) that causes an executable to run with the owner's privileges rather than the caller's. Only `passwd` and `sudo`-class binaries should have this.

**SetGID** — Same as SetUID but applies the group's permissions instead of the owner's.

**ACL (Access Control List)** — A fine-grained permission system beyond rwx owner/group/other. Used in the labs with `setfacl`/`getfacl` to grant the CloudWatch agent read access to audit logs without weakening directory security.

**setfacl / getfacl** — Commands to set and view ACL permissions on files and directories.

**/etc/shadow** — The file containing hashed Linux user passwords. Readable only by root. A common credential dumping target.

**crontab** — A per-user cron schedule file. Attackers use cron jobs for persistence — code that re-executes on a schedule even after a reboot.

**nginx** — A widely-used open-source web server and reverse proxy. Used in the labs to generate HTTP telemetry.

**chmod** — Change file permissions. Absolute mode (`chmod 755`) or relative mode (`chmod o-rwx`).

**File permissions (rwx)** — Linux permission model: read (4), write (2), execute (1). Applied to owner, group, and others. Common modes: 600 (private key), 644 (config file), 755 (executable).

**Privilege escalation** — Gaining higher access rights than originally authorized, typically from a standard user to root.

**Reverse shell** — A network connection initiated outbound from a compromised host to an attacker's listener, bypassing inbound firewall rules.

**Backdoor** — A persistent, often hidden access mechanism installed by an attacker to regain access after initial compromise.

## AWS Core Services

**EC2 (Elastic Compute Cloud)** — AWS's virtual machine service. The `soc-lab` instance is an Amazon Linux 2023 EC2 instance.

**AMI (Amazon Machine Image)** — A pre-configured virtual machine image used to launch EC2 instances.

**Security group** — An EC2 virtual firewall that controls inbound and outbound traffic by port, protocol, and IP range.

**VPC (Virtual Private Cloud)** — An isolated network environment in AWS where EC2 instances and other resources run.

**IAM (Identity and Access Management)** — AWS's service for managing users, roles, policies, and permissions.

**IAM role** — An AWS identity with attached policies that can be assumed by services (EC2, Lambda) or users. No long-term credentials.

**IAM policy** — A JSON document defining permissions. Attached to users, groups, or roles.

**Instance profile** — A container for an IAM role that allows EC2 instances to assume that role and make AWS API calls.

**Trust policy** — The part of an IAM role definition that specifies who or what can assume the role (`Principal`).

**AdministratorAccess** — An AWS-managed policy granting full access to all AWS services and resources. A critical finding when attached to IAM users.

**Least-privilege policy** — An IAM policy granting only the specific actions on specific resources required for the task.

**sts:AssumeRole** — The IAM action that allows an entity to assume a role and receive temporary credentials.

**STS (Security Token Service)** — AWS service that issues temporary, scoped credentials when a role is assumed. `sts get-caller-identity` returns the current identity.

**Credential report** — An IAM-generated CSV listing all IAM users and their credential status (MFA, access keys, last used). Used to find users without MFA.

**Root account** — The original AWS account owner with unrestricted access to all services. Should have MFA enabled and access keys deleted.

**CloudShell** — AWS's browser-based shell with pre-authenticated CLI access. No local setup required.

## AWS Security Services

**GuardDuty** — AWS's managed threat detection service. Uses ML and threat intelligence to detect anomalous activity in logs, network traffic, and runtime behavior.

**GuardDuty Runtime Monitoring** — A GuardDuty feature that deploys a lightweight agent to EC2 instances via SSM to detect OS-level threats: reverse shells, privilege escalation, unusual process activity.

**Security Hub** — AWS's centralized security findings aggregation service. Integrates with GuardDuty, Config, and Inspector. Supports CIS Benchmark and AWS Foundational Security Best Practices standards.

**CIS Benchmark (AWS)** — Center for Internet Security standards for securely configuring AWS. Applied in Security Hub as a compliance standard.

**FSBP (AWS Foundational Security Best Practices)** — AWS's own set of security controls evaluated by Security Hub.

**AWS Config** — A service that records AWS resource configurations and evaluates them against compliance rules. Required for Security Hub to detect configuration drift continuously.

**CloudTrail** — AWS's audit log for API calls. Records who called what AWS API, when, and from where. Event History retains 90 days with no configuration; a Trail extends this indefinitely to S3.

**CloudTrail trail** — A configuration that continuously delivers CloudTrail events to an S3 bucket (and optionally CloudWatch Logs), enabling long-term retention and alerting.

**SSM (AWS Systems Manager)** — A fleet management service for EC2. Used in the labs to deploy the GuardDuty runtime agent. The `AmazonSSMManagedInstanceCore` policy enables SSM access.

**S3 (Simple Storage Service)** — AWS's object storage service. Used in the labs to create a public-access finding and as a Lambda permission target.

**SNS (Simple Notification Service)** — AWS's pub/sub messaging service. Used to send alarm notifications (email, Lambda trigger) when a CloudWatch alarm fires.

**SNS topic** — A named channel in SNS to which publishers send messages and subscribers receive them.

**Lambda** — AWS's serverless compute service. Executes code in response to events without provisioning servers. Referenced in the labs as an IAM permission target.

**Bedrock** — AWS's managed large language model service. Supports invoking foundation models via API. Referenced in the labs as an IAM permission target for Lambda.

## CloudWatch

**CloudWatch Logs** — AWS's log ingestion and storage service. The `/soc-lab/secure` log group stores auditd events shipped from EC2.

**Log group** — A named container for log streams in CloudWatch Logs. Equivalent to a log source.

**Log stream** — A sequence of log events from a single source within a log group (typically one per EC2 instance ID).

**CloudWatch agent (`amazon-cloudwatch-agent`)** — AWS software installed on EC2 instances to ship OS logs to CloudWatch Logs. Configured with a JSON file. Runs as the `cwagent` system user.

**Logs Insights** — CloudWatch's query language for analyzing log events. Supports `fields`, `filter`, `parse`, `stats`, `sort`, `limit`, and `display` commands.

**Log Analytics** — A CloudWatch console feature providing an AI-powered natural language interface over Logs Insights. Ask a question in English; it generates the query.

**Metric filter** — A CloudWatch rule that scans log events for a pattern and increments a custom metric counter on each match.

**Filter pattern** — The search expression used in a metric filter (e.g., `sudo res=failed`). CloudWatch uses its own simple pattern syntax, not regex in basic filters.

**Metric namespace** — A grouping container for custom CloudWatch metrics (e.g., `SOCLab`).

**CloudWatch alarm** — A monitoring rule that transitions between states (OK / In alarm / Insufficient data) based on a metric threshold. Can trigger SNS notifications and AI Operations investigations.

**Alarm state** — One of three states: `OK` (metric within threshold), `In alarm` (threshold breached), `Insufficient data` (not enough data points to evaluate).

**Dashboard widget** — A visualization component on a CloudWatch dashboard. Types used in the labs: line chart (metrics over time), bar chart (aggregated counts), logs table (raw event rows).

**CloudWatch AI Operations** — A CloudWatch feature that automatically creates AI-powered investigations when alarms fire or investigations are manually triggered. Generates hypotheses about anomalies.

**Investigation (AI Operations)** — An AI Operations construct that groups related events, generates hypotheses, and provides an analyst interface for triage and closure.

**Query Generator** — The "Ask AI to write a query" interface in CloudWatch Log Analytics and the investigation console, which generates Logs Insights syntax from natural language prompts.

## Data and Formats

**JSONL (JSON Lines)** — A text file format where each line is a valid JSON object. Used throughout the labs for log export and processing.

**Auditd log format** — Space-separated key=value pairs preceded by `type=TYPENAME`. Example: `type=USER_CMD uid=1000 auid=1000 ... cmd=...`

**CSV (Comma-Separated Values)** — A plain-text tabular format. Used in the labs for findings output and posture baselines.

**Log normalization** — Converting raw, heterogeneous log formats into a consistent schema for easier analysis and correlation.

**Schema** — The defined structure of a data record: its field names, types, and meanings.

## Cloud Misconfigurations

**Misconfiguration** — An incorrect or insecure cloud resource setting. Common examples: SSH open to 0.0.0.0/0, MFA disabled, public S3 bucket.

**Public access block** — S3 controls that prevent public access via ACLs or bucket policies. All four settings should be `true` for non-public buckets.

**Open security group** — A security group allowing inbound traffic from `0.0.0.0/0` on sensitive ports (e.g., 22/SSH). A CRITICAL/HIGH finding.

**Posture** — An overall assessment of how secure a cloud environment is across identity, network, data, and logging controls.

**Remediation** — A corrective action taken to fix a misconfiguration or vulnerability. In the labs, narrowing SSH CIDR from `0.0.0.0/0` to your IP.

**CIDR** — Classless Inter-Domain Routing. A notation for IP address ranges (e.g., `0.0.0.0/0` = all IPs; `203.0.113.1/32` = one specific IP).

## AI-Assisted Security Workflows

**AI coding assistant** — A chat-based AI tool (Amazon Q, Claude, ChatGPT) used to generate, review, and improve code. Used in the labs for generating log parsers and IAM policies.

**Prompt engineering** — The practice of writing clear, structured inputs to AI systems to get accurate, safe, and useful outputs.

**Human-in-the-loop** — A workflow design where a human reviews and approves AI outputs before they are executed or deployed. Critical for security use cases.

**AI-generated code review** — Evaluating AI output against security criteria before execution: no shell injection, no hardcoded secrets, safe error handling.

**Severity hint** — A label assigned by a script or model indicating the relative risk of an event. Should be explainable from the data, not assigned arbitrarily.

## General Security Architecture

**SIEM (Security Information and Event Management)** — A platform that aggregates, normalizes, and correlates security events from many sources. Enables alerting, dashboards, and long-term forensic search. CloudWatch Logs + AI Operations is a simplified SIEM-like pipeline.

**EDR (Endpoint Detection and Response)** — Security software installed on endpoints to detect, record, and respond to threats at the OS level. GuardDuty Runtime Monitoring is an AWS-native EDR equivalent.

**XDR (Extended Detection and Response)** — Integrates EDR with network, cloud, and email telemetry for cross-domain threat correlation.

**Defense in depth** — A layered security strategy where multiple independent controls must all be bypassed for an attack to succeed.

**Zero Trust** — A security model that treats every access request as untrusted by default, requiring continuous verification regardless of network location.

**Attack surface** — The total set of paths through which an attacker could gain unauthorized access to a system.

**Attack vector** — A specific entry path used to breach a system (e.g., open SSH port, misconfigured IAM role).

**Lateral movement** — Moving from an initial foothold to other systems within the same environment, expanding access.

**Persistence** — Mechanisms used by attackers to maintain access after a reboot or credential reset (e.g., cron jobs, new accounts, SSH authorized keys).

**Exfiltration** — Transferring stolen data from the victim environment to attacker-controlled infrastructure.

**Command and Control (C2)** — Communication channels used by malware or an attacker to send instructions to a compromised host and receive results.

**Post-exploitation** — Actions taken after initial access is achieved: privilege escalation, persistence, lateral movement, data collection.

**Reconnaissance** — The attacker's information-gathering phase: enumerating open ports, cloud services, user accounts, and software versions.

**Intrusion detection** — Automated monitoring for signs of unauthorized access or policy violations. CloudWatch alarms + metric filters are a form of signature-based intrusion detection.
