## Lab 2.3: Attack and Detect Capstone

This lab puts you on both sides of the wire. You will pick one threat from a menu, build a dedicated detection pipeline for it — metric filter, alarm, and dashboard widget — then simulate the actual attack and watch everything fire. The AI Operations investigation you built into the pipeline runs automatically. Your final deliverable is an incident report closing that investigation.

This is a synthesis exercise. You have seen every component before. Now you own the full loop.

## Attacker Objective (Cyber Lens)

Objective: Execute a realistic attack technique against your own system and verify end-to-end detection without manually triggering any alert.
ATT&CK focus: Determined by the threat you select.
Defender outcome: Confirm that a raw log event flows through auditd → CloudWatch → metric filter → alarm → AI investigation with no manual intervention at any stage.

## Learning Objectives

By the end of this lab, you will be able to:

1. Select a threat and independently translate it into a working CloudWatch detection.
2. Build a metric filter, alarm, and dashboard widget without step-by-step guidance.
3. Simulate an attack technique on a live Linux endpoint.
4. Evaluate an auto-triggered AI Operations investigation and select the most credible hypothesis.
5. Write a structured incident report and formally close an investigation.

## Part 3: Build the Detection Pipeline

Build a dedicated detection pipeline for your chosen threat. You can make use of auditd rules on commands such as

### 2a. Create a Metric Filter

1. Navigate to **CloudWatch** → **Logs** → **Log groups** → **/soc-lab/audit**.
2. Open the **Metric filters** tab and click **Create metric filter**.
3. Enter the filter pattern from the table above for your chosen threat.
4. Test the pattern against recent log data. If there are no matching events yet, move to the simulation step first (Part 3), generate the events, then come back and re-test.
5. On the **Assign metric** page, set:
   - **Metric namespace**: `SOCLab`
   - **Metric name**: a short, descriptive name for your threat (e.g., `SSHBruteForce`, `NewAccountCreated`, `ShadowFileAccess`, `SuAbuse`, `CronInstalled`)
   - **Metric value**: `1`
   - **Default value**: `0`
6. Save the filter.

![Create metric filter on a CloudWatch log group](images/create-log-filter.png)

### 2b. Create the Alarm

1. Navigate to **CloudWatch** → **Alarms** → **Create alarm** → **Select metric** → **SOCLab** → select your new metric.
2. Configure:
   - **Statistic**: Sum
   - **Period**: 1 minute
   - **Condition**: Greater than or equal to `1`
3. Add an **AI Operations investigation** action triggered on **In alarm** state. This is the same action type used in Lab 2.2.
4. Name the alarm something descriptive (e.g., `ssh-brute-force-alarm`).

![Hovering over a metric widget reveals the bell icon to create an alarm](images/create-alarm.png)

### 2c. Add a Dashboard Widget

1. Open the `soc-lab-web-monitor` dashboard.
2. Click **Add widget**. Follow **Path B** (Logs → Logs table) from Lab 2.2 Part 3 — select **Logs** as the data type, not Metrics.
3. Set log group to **`/soc-lab/audit`** and paste the Logs Insights query for your chosen threat below. Each query filters by your metric filter pattern and displays `@timestamp` and `@message`.
4. Save the dashboard.

**Logs table** (event detail) — recommended for all threats:

**Threat A — SSH brute force**

```
fields @timestamp, @message
| filter @message like 'exe="/usr/sbin/sshd"' and @message like 'res=failed'
| sort @timestamp desc
| limit 50
```

Title: `SSH Brute Force Events`

**Threat B — New account backdoor**

```
fields @timestamp, @message
| filter @message like 'type=ADD_USER'
| sort @timestamp desc
| limit 50
```

Title: `New Account Events`

**Threat C — Sensitive file access**

```
fields @timestamp, @message
| filter @message like 'name="/etc/shadow"'
| sort @timestamp desc
| limit 50
```

Title: `Shadow File Access`

**Threat D — `su` abuse**

```
fields @timestamp, @message
| filter @message like 'exe="/usr/bin/su"' and @message like 'res=failed'
| sort @timestamp desc
| limit 50
```

Title: `Su Abuse Events`

**Threat E — Cron persistence**

```
fields @timestamp, @message
| filter @message like 'crontab'
| sort @timestamp desc
| limit 50
```

Title: `Cron Persistence Events`

**Line chart** (count over time) — optional alternative. Replace the `filter` line with the one from your threat above:

```
fields @timestamp
| filter @message like 'exe="/usr/sbin/sshd"' and @message like 'res=failed'
| stats count(*) as events by bin(5m)
```

## Part 4: Simulate the Attack

SSH into your EC2 instance and run the simulation commands for your chosen threat. Run the `testattacker` user simulation steps from a separate terminal session logged in as `testattacker` (use `su - testattacker` or a second SSH session with that account).

---

### Threat A — SSH Brute Force

From your **local machine** (not EC2), run repeated failed SSH login attempts. Replace `<EC2-IP>` with your instance's public IP:

```bash
for i in $(seq 1 15); do
  ssh -o StrictHostKeyChecking=no -o ConnectTimeout=3 wronguser@<EC2-IP> 2>/dev/null
  sleep 1
done
```

Each connection attempt will fail and generate an audit record with `exe="/usr/sbin/sshd" res=failed` in `/var/log/audit/audit.log`, which the CloudWatch agent ships to the `/soc-lab/audit` log group.

---

### Threat B — New Account Backdoor

Run as the `ec2-user` (your normal admin account, which has sudo):

```bash
sudo useradd -m backdoor-user
```

This creates an `ADD_USER` audit event. To clean up after the lab:

```bash
sudo userdel -r backdoor-user
```

---

### Threat C — Sensitive File Access

As `testattacker` (no sudo), access the shadow file directly:

```bash
cat /etc/shadow
```

The read will likely be denied (`Permission denied`), but auditd still records the access attempt when an audit rule watches that file. If the audit rule is not configured, add it first:

> **Run the next command as `ec2-user`** (the account with sudo), not as `testattacker`. After the rule is loaded, switch back to `testattacker` to run the `cat` command.

```bash
sudo auditctl -w /etc/shadow -p r -k shadow-access
```

Then retry the `cat` as `testattacker`.

---

### Threat D — `su` Abuse

As `testattacker`, repeatedly attempt to switch to root with a wrong password:

```bash
for i in $(seq 1 10); do
  su - root
  sleep 2
done
```

When prompted for a password, type an incorrect password and press Enter. Repeat until all 10 iterations complete. Each failure generates an audit record with `exe="/usr/bin/su" res=failed` in `/var/log/audit/audit.log`.

> **Note**: `su` reads the password from the terminal directly and cannot be driven by piping text to stdin on Amazon Linux 2023. Enter the wrong password interactively at each prompt.

---

### Threat E — Cron Persistence

As `testattacker`, install a cron entry:

```bash
(crontab -l 2>/dev/null; echo "*/5 * * * * /bin/bash -i") | crontab -
```

Then verify it was written:

```bash
crontab -l
```

To remove after the lab:

```bash
crontab -r
```

## Part 5: Verify the Pipeline Fired

After running the simulation, work through each stage of the pipeline to confirm end-to-end detection.

1. **Log group** — In CloudWatch → Logs → Log groups → /soc-lab/audit, open the most recent log stream. Find a log event that matches your threat pattern. Copy the raw event text — you will use it in your incident report.

2. **Metric** — Navigate to **CloudWatch** → **Metrics** → **SOCLab** and find your metric. Confirm the count incremented in the time window when you ran the simulation.

3. **Alarm** — Navigate to **CloudWatch** → **Alarms** and confirm your alarm transitioned to **In alarm** state during the simulation window. If it is still showing **Insufficient data**, wait two minutes and refresh — the 1-minute period needs to complete before evaluation.

4. **Dashboard** — Open `soc-lab-web-monitor`. Your new widget should show the spike. If the dashboard time range does not capture the event, adjust it to cover the last 30 minutes.

5. **AI Investigation** — Navigate to **CloudWatch** → **AI Operations** → **Investigations**. An investigation should have been created automatically when the alarm fired. Open it.

## Part 6: Evaluate the AI Investigation

The investigation was opened automatically by the alarm action. CloudWatch AI Operations has already run initial analysis. Your job is analyst triage.

1. Read the **Summary** at the top of the investigation. Note what the AI identified as the triggering signal.

2. Review the **Contributing factors** or **Hypotheses** panel. The AI will offer candidate explanations for the anomaly — for example, "unusual account activity" or "authentication failure spike."

3. Select the hypothesis that best matches what you actually did. If there is a free-text notes field, briefly describe the simulation you ran.

4. Check whether the AI linked any related log events, metrics, or other alarms. Note anything that was not linked but should have been.

5. Do **not** close the investigation yet — you will close it after writing the incident report in Part 6.

---

## Part 7: Write the Incident Report

Write a brief incident report in the investigation notes or in a separate document. Use the structure below. Keep it factual and concise — this is a triage report, not an essay.

---

**Incident Report — [Your Name] — [Date]**

**Threat simulated:** [Threat letter and name from the menu]

**ATT&CK technique:** [Technique ID and name]

**Timeline:**
- [HH:MM UTC] — Simulation started
- [HH:MM UTC] — First matching log event observed in CloudWatch
- [HH:MM UTC] — Alarm transitioned to In Alarm
- [HH:MM UTC] — AI investigation opened automatically

**Raw log evidence:**
Paste one representative log event from Step 1 of Part 5.

**Detection gap (if any):**
Did anything in the pipeline fail to fire, fire late, or produce a false hypothesis? Describe briefly, or write "None — pipeline fired as expected."

**Recommended follow-up:**
What would you do next if this were a real incident? (One or two sentences.)

**Disposition:** Simulated / Closed — no further action required.

---

After writing the report, return to the investigation and click **Close investigation** (or the equivalent action in the AI Operations console). Select the appropriate closure reason — typically "Investigated — no action required" for a simulation.
