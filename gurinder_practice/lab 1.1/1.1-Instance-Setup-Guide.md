## Key Concepts (Theory)

Read this section once before you start. You do not need to memorize everything — come back here when a step mentions a term you do not recognize.

### 1. AWS Account & Region

- **AWS account** — Your personal cloud workspace. You own the resources you create and pay for what you use.
- **Region** — A physical geographic area (e.g. `us-east-1` = N. Virginia). Resources in one region do not automatically exist in another.
- **Why us-east-1?** The lab AMI (machine image) ID is tied to this region. Always select **US East (N. Virginia)** in the top-right of the AWS Console.

### 2. EC2 (Elastic Compute Cloud)

An **EC2 instance** is a virtual machine (VM) running in AWS. Think of it as a computer you rent by the hour:

| Term | Plain English |
|------|---------------|
| **Instance** | Your virtual server |
| **AMI** | A template snapshot of an OS (like a Windows/Linux install DVD) |
| **Instance type** | Size/CPU/RAM (lab uses `t2.medium`) |
| **Public IP** | Address address on the internet so you can reach the server |

- **VPC** — Isolated virtual network. Your EC2 instance lives inside it.
- **Subnet** — A subdivision of the VPC (often one per availability zone).
- **Security Group (SG)** — A stateful firewall attached to your instance. You explicitly allow ports (22 for SSH, 80 for HTTP).

> **Security note:** Opening SSH (22) to `0.0.0.0/0` means *anyone on the internet* can *try* to connect. Key-based auth protects you, but for production you would restrict to your home IP. For this training lab on a disposable instance, it is acceptable.

| `soc-lab-key.pem` | **Private key** — stays on your machine only |
| Public key (in AWS) | Paired with the private key; AWS installs it on the instance at launch |

**VS Code Remote - SSH** uses this key to open a full remote workspace — terminal, file explorer, and editor — as if the server were local.

### 5. Linux File Permissions (Preview)

Linux represents permissions as three groups: **owner**, **group**, **others**. Each gets `r` (read), `w` (write), `x` (execute).

Example: `-rwxr-xr-x` → owner can read/write/execute; group and others can read/execute only.

You will practice this in the [Terminal Basics](#terminal-basics) section.

### 6. SetUID (Preview)

Some programs must run with elevated privileges (e.g. `passwd` changing passwords). **SetUID** marks the program to run as the *file owner* (often `root`) no matter who launched it. Attackers hunt misconfigured SetUID binaries for privilege escalation — defenders inventory them.

### 7. nginx & Telemetry

**nginx** is a lightweight web server. Every HTTP request creates a line in `/var/log/nginx/access.log`. Later AI-assisted detection labs need this **telemetry** — without logs and services, there is nothing to investigate.

### Step A — Create an SSH Key Pair

You need a key **before** launching the instance so AWS can install the matching public key on the server.

#### Option 1: AWS CloudShell (recommended for beginners)

1. Log in to [AWS Console](https://console.aws.amazon.com/).
2. Confirm region = **us-east-1** (top-right).
3. Click the **CloudShell** icon (terminal prompt `>_` in the top navigation bar).
4. Run:

```bash
aws ec2 create-key-pair \
  --key-name soc-lab-key \
  --query 'KeyMaterial' \
  --output text > soc-lab-key.pem

chmod 400 soc-lab-key.pem
```

5. **Download the key to your PC:** CloudShell → **Actions** → **Download file** → choose `soc-lab-key.pem`.
6. Save it to a known location, e.g. `C:\Users\GURINDER\Downloads\soc-lab-key.pem`.

> **Screenshot:** Save proof of key creation in `lab 1.1 screenshots/`.

#### Option 2: AWS Console (no CLI)

1. EC2 → **Key Pairs** → **Create key pair**
2. Name: `soc-lab-key`, type: **RSA**, format: **.pem**
3. Download automatically starts — store the file safely.

> If you use the Console method, skip `--key-name soc-lab-key` in later CLI steps; the name is already registered.

---

### Step B — Create Security Group & EC2 Instance

Run these in **CloudShell** (still in `us-east-1`). Paste the whole block, then press **Enter** after the last line.

```bash
# 1. Create security group (firewall)
aws ec2 create-security-group \
  --group-name soc-lab-sg \
  --description "SOC Lab SG" || true

# 2. Find default VPC
VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' \
  --output text)

# 3. Get security group ID
SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=soc-lab-sg Name=vpc-id,Values=$VPC_ID \
  --query 'SecurityGroups[0].GroupId' \
  --output text)

# 4. Allow SSH (port 22)
aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp \
  --port 22 \
  --cidr 0.0.0.0/0 || true

# 5. Allow HTTP (port 80)
aws ec2 authorize-security-group-ingress \
  --group-name soc-lab-sg \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0 || true

# 6. Pick a default subnet
SUBNET_ID=$(aws ec2 describe-subnets \
  --filters Name=vpc-id,Values=$VPC_ID Name=default-for-az,Values=true \
  --query 'Subnets[0].SubnetId' \
  --output text)

# 7. Launch EC2 instance
INSTANCE_ID=$(aws ec2 run-instances \
  --image-id ami-098e39bafa7e7303d \
  --instance-type t2.medium \
  --key-name soc-lab-key \
  --network-interfaces "AssociatePublicIpAddress=true,DeviceIndex=0,SubnetId=$SUBNET_ID,Groups=$SG_ID" \
  --query 'Instances[0].InstanceId' \
  --output text)

echo "Instance ID: $INSTANCE_ID"
```

**What each piece does (quick reference):**

| Command block | Purpose |
|---------------|---------|
| `create-security-group` | Creates firewall object |
| `authorize-security-group-ingress` | Opens ports 22 and 80 |
| `run-instances` | Launches Amazon Linux 2023 server with your SSH key |

> **Screenshot:** EC2 → Instances showing **running** state and instance ID.

#### Console alternative (launch wizard)

If CLI feels overwhelming:

1. **EC2 → Launch Instance**
2. Name: `soc-lab-instance`
3. AMI: **Amazon Linux 2023** (same as `ami-098e39bafa7e7303d` in us-east-1)
4. Type: **t2.medium**
5. Key pair: **soc-lab-key**
6. Network: allow **SSH** and **HTTP** from anywhere (0.0.0.0/0)
7. Launch

### Step C — Get the Public IP Address

Wait **30–60 seconds** after launch, then:

```bash
aws ec2 describe-instances \
  --instance-ids $INSTANCE_ID \
  --query 'Reservations[0].Instances[0].PublicIpAddress' \
  --output text
```

**Write down your public IP** — you need it for SSH and the browser test.

> **Tip:** CloudShell variables disappear if the session resets. If `$INSTANCE_ID` is empty, find the IP in **EC2 → Instances → Public IPv4 address**.

> **Screenshot:** Public IP visible in console or terminal output.

---

### Step D — Configure VS Code for SSH

1. Open **VS Code** on your Windows PC.
2. Open the **Remote Explorer** sidebar (monitor/remote icon, or `Ctrl+Shift+P` → "Remote-SSH: Open SSH Configuration File").
3. Click **SSH** → gear icon → select the **User** config file (not workspace).
4. Replace contents with (update paths and IP):

```ssh-config
Host SOC-Instance
  HostName YOUR_PUBLIC_IP_HERE
  IdentityFile "C:\Users\GURINDER\Downloads\soc-lab-key.pem"
  User ec2-user
```

5. **Save** the file.

| Setting | Meaning |
|---------|---------|
| `HostName` | Your EC2 public IP |
| `IdentityFile` | Full path to `.pem` on **your** PC |
| `User` | Default Amazon Linux user is always `ec2-user` |

---

### Step E — Connect & Verify

1. In Remote Explorer, hover **SOC-Instance** → click the **→** connect arrow.
2. First connect may ask about host fingerprint — choose **Continue**.
3. Bottom-left corner turns **blue/green** and shows `SSH: SOC-Instance` when connected.
4. **File → Open Folder** → type `/home/ec2-user` → OK.
5. **Terminal → New Terminal** — you are now on the remote Linux server.

Verify:

```bash
hostnamectl
whoami
pwd
```

Expected: `whoami` prints `ec2-user`; `pwd` is `/home/ec2-user`.

## SetUID & Privilege Escalation Risk

### Inspect a known SetUID binary

```bash
ls -l /usr/bin/passwd
```

Look for **`s`** in the permission slot (e.g. `-rwsr-xr-x`). That means: runs as **root** when any user executes it.

### Find all SetUID binaries

```bash
find / -perm -4000 -exec ls -l {} \; 2>/dev/null
```

This may take 1–2 minutes. You should see `/usr/bin/sudo` in the list.

## Run a Basic Web Service (nginx)

This section is **required** — later labs depend on nginx logs existing.

### Step 1 — Install and start nginx

```bash
sudo dnf install -y nginx
sudo systemctl enable --now nginx
sudo systemctl status nginx --no-pager
```

Look for **`active (running)`** in green.

### Step 2 — Replace default page

```bash
cat <<'EOF' | sudo tee /usr/share/nginx/html/index.html
<!doctype html>
<html>
  <head><title>SOC Lab Service</title></head>
  <body>
    <h1>SOC Lab Service Running</h1>
    <p>Training service endpoint active.</p>
  </body>
</html>
EOF
```

### Step 3 — Test from server and browser

On the server:

```bash
curl -I http://localhost
curl -s http://localhost | head -n 5
```

On your **Windows PC**, open a browser:

```text
http://YOUR_PUBLIC_IP
```

You should see **SOC Lab Service Running**.

> **Screenshot:** Browser showing the lab page → `lab 1.1 screenshots/`.

### Step 4 — Generate log traffic

```bash
for i in {1..5}; do curl -s http://localhost >/dev/null; done
sudo tail -n 20 /var/log/nginx/access.log
```

Each line in `access.log` is one HTTP request — this is **telemetry** for future detection labs.

## Troubleshooting

### SSH connection fails / times out

| Check | Action |
|-------|--------|
| Instance state | Must be **running** |
| Public IP | Confirm IP has not changed (stop/start can change it) |
| Security group | Port **22** open to your IP or `0.0.0.0/0` |
| Key path | `IdentityFile` path must match where `.pem` actually lives |
| Key permissions | On Windows, path in quotes is usually enough; WSL may need `chmod 400` |

### `Permission denied (publickey)`

- Wrong `.pem` file or wrong key pair on instance
- Wrong user — Amazon Linux uses `ec2-user`, not `ubuntu` or `root`

### Browser cannot load `http://PUBLIC_IP`

- Security group must allow **port 80**
- nginx must be running: `sudo systemctl start nginx`
- Try `curl http://localhost` on the server first — if that works, issue is firewall/IP

### CloudShell says `InvalidAMIID.NotFound`

- You are in the wrong region. switch to **us-east-1**

### Variables lost in CloudShell

Re-fetch instance ID from the console, or:

```bash
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].[InstanceId,PublicIpAddress,Tags[?Key==`Name`].Value|[0]]' \
  --output table
```
