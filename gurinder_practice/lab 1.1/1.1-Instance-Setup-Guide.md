## Key Concepts

### AWS & EC2

| Term | Meaning |
|------|---------|
| **AWS account** | Your personal cloud workspace — you own resources and pay for usage |
| **Region** | Geographic area (`us-east-1` = N. Virginia). Resources do not cross regions automatically |
| **EC2 instance** | A virtual server (VM) in the cloud |
| **AMI** | OS template used to launch the instance |
| **Instance type** | CPU/RAM size — this lab uses `t2.medium` |
| **Public IP** | Internet address to reach your server |

> Always set the AWS Console region to **us-east-1**. The lab AMI ID only works there.

|------|---------|

**SSH (Secure Shell)** opens a remote terminal. **VS Code Remote - SSH** extends that into a full remote workspace (editor, files, terminal).

### Linux Permissions & SetUID *(covered hands-on in Phase 3)*

| Topic | Summary |
|-------|---------|
| **Permissions** | Three groups (owner, group, others) × three flags (`r` read, `w` write, `x` execute) |
| **SetUID** | Program runs as file owner (often `root`) regardless of who launched it — attackers hunt these for privilege escalation |

### nginx & Telemetry *(covered hands-on in Phase 4)*

**nginx** serves HTTP on port 80. Each request writes a line to `/var/log/nginx/access.log`. Later detection labs need this telemetry — without logs, there is nothing to investigate.

### Step B — Create Security Group & EC2 Instance

| | |
|---|---|
| **Goal** | Launch a Linux server with ports 22 and 80 open |
| **Time** | ~10 min |
| **Screenshot** | EC2 instance in **running** state → `lab 1.1 screenshots/` |

<details>
<summary><strong>CloudShell — paste entire block</strong></summary>

```bash
# Create security group
aws ec2 create-security-group \
  --group-name soc-lab-sg \
  --description "SOC Lab SG" || true

# Find default VPC
VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' \
  --output text)

# Get security group ID
SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=soc-lab-sg Name=vpc-id,Values=$VPC_ID \
  --query 'SecurityGroups[0].GroupId' \
  --output text)

# Allow SSH (22) and HTTP (80)
aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID --protocol tcp --port 22 --cidr 0.0.0.0/0 || true

aws ec2 authorize-security-group-ingress \
  --group-name soc-lab-sg --protocol tcp --port 80 --cidr 0.0.0.0/0 || true

# Default subnet
SUBNET_ID=$(aws ec2 describe-subnets \
  --filters Name=vpc-id,Values=$VPC_ID Name=default-for-az,Values=true \
  --query 'Subnets[0].SubnetId' \
  --output text)

# Launch instance
INSTANCE_ID=$(aws ec2 run-instances \
  --image-id ami-098e39bafa7e7303d \
  --instance-type t2.medium \
  --key-name soc-lab-key \
  --network-interfaces "AssociatePublicIpAddress=true,DeviceIndex=0,SubnetId=$SUBNET_ID,Groups=$SG_ID" \
  --query 'Instances[0].InstanceId' \
  --output text)

echo "Instance ID: $INSTANCE_ID"
```

| Command | Purpose |
|---------|---------|
| `create-security-group` | Creates firewall object |
| `authorize-security-group-ingress` | Opens ports 22 and 80 |
| `run-instances` | Launches Amazon Linux 2023 with your SSH key |

</details>

<details>
<summary><strong>Console alternative — Launch wizard</strong></summary>

1. **EC2 → Launch Instance**
2. Name: `soc-lab-instance`
3. AMI: **Amazon Linux 2023**
4. Type: **t2.medium** · Key: **soc-lab-key**
5. Network: allow **SSH** and **HTTP** from `0.0.0.0/0`
6. Launch

</details>

**Verify:** Instance shows **running** in EC2 → Instances.

### Step D — Configure VS Code SSH

| | |
|---|---|
| **Goal** | Point VS Code at your instance with the correct key |
| **Time** | ~5 min |

1. Open **VS Code**
2. **Remote Explorer** → **SSH** → gear icon → **User** config file  
   *(Or `Ctrl+Shift+P` → "Remote-SSH: Open SSH Configuration File")*
3. Add (replace IP and path if needed):

```ssh-config
Host SOC-Instance
  HostName YOUR_PUBLIC_IP_HERE
  IdentityFile "C:\Users\GURINDER\Downloads\soc-lab-key.pem"
  User ec2-user
```

| Setting | Meaning |
|---------|---------|
| `HostName` | EC2 public IP from Step C |
| `IdentityFile` | Full path to `.pem` on your PC |
| `User` | Always `ec2-user` for Amazon Linux |

4. Save the file.

## Troubleshooting

<details>
<summary><strong>SSH connection fails or times out</strong></summary>

| Check | Action |
|-------|--------|
| Instance state | Must be **running** |
| Public IP | May change after stop/start — confirm current IP |
| Security group | Port **22** open |
| Key path | `IdentityFile` must match actual `.pem` location |

</details>

<details>
<summary><strong>Permission denied (publickey)</strong></summary>

- Wrong `.pem` or wrong key pair on the instance
- Wrong user — use `ec2-user`, not `ubuntu` or `root`

</details>

<details>
<summary><strong>Browser cannot load http://PUBLIC_IP</strong></summary>

- Security group must allow port **80**
- nginx running: `sudo systemctl start nginx`
- Test on server first: `curl http://localhost` — if that works, issue is firewall or IP

</details>

<details>
<summary><strong>InvalidAMIID.NotFound</strong></summary>

- Wrong AWS region — switch to **us-east-1**

</details>

<details>
<summary><strong>CloudShell variables lost</strong></summary>

```bash
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].[InstanceId,PublicIpAddress,Tags[?Key==`Name`].Value|[0]]' \
  --output table
```

</details>
