## The story

You are building a **secure lab building** in AWS — a Linux server you will use for all later cybersecurity labs. Think of it act by act:

**Act 1 — Forge your personal key (Steps 1)**  
Before the building exists, you create a **key pair**. The private key stays on your Windows PC; AWS keeps the public half. No key = no entry.  
→ *Step 1 — Create SSH key*

**Act 2 — Construct the building with a bouncer (Step 2)**  
You launch an **EC2 instance** inside a **VPC** (your cloud network). A **Security Group** acts as a bouncer: only **port 22** (SSH) and **port 80** (web) get in.  
→ *Step 2 — Launch EC2*

**Act 3 — Put an address on the building (Step 3)**  
The server gets a **public IP** — its street address on the internet so your PC can find it.  
→ *Step 3 — Get public IP*

**Act 4 — Program your key fob on Windows (Step 4)**  
You tell **VS Code** which address and key file to use. On Windows, you also **lock down** the `.pem` file so only you can read it — like a key fob nobody else can copy.  
→ *Step 4 — VS Code SSH config & key permissions*

**Act 5 — Walk through the front door (Step 5)**  
You **SSH in** from VS Code. You are now inside the building with a remote terminal — your workspace for every lab after this.  
→ *Step 5 — Connect to EC2*

**Act 6 — Run a tool on a lab computer (Step 6)**  
Inside the server, you create a small **script** and use **chmod** to make it executable — basic Linux skills every SOC analyst needs.  
→ *Step 6 — Script & chmod*

**Act 7 — Find programs with hidden master-key power (Step 7)**  
Some programs can run as **root** even when a normal user starts them (**SetUID**). You hunt for these — they matter for privilege-escalation detection.  
→ *Step 7 — SetUID find*

**Act 8 — Open the storefront and start a visitor log (Step 8)**  
You install **nginx** (web server). Visitors (or attackers in Lab 1.2) hit port 80; every request writes a line to **access.log** — **telemetry** for future detection labs.  
→ *Step 8 — nginx & browser test*

**What you end up with**

| Piece | Story role | Real name |
|-------|------------|-----------|
| Your Windows PC | You, with the key | Local machine + VS Code |
| EC2 instance | The building | Amazon Linux server |
| Security Group | Bouncer at the door | Virtual firewall |
| `.pem` file | Your private key | SSH key pair |
| nginx + access.log | Visitor sign-in book | Web telemetry |

## Step 1 — Create SSH key (Block 1)

**Story:** Act 1 — forge your personal key before the building exists.  
**Terms:** SSH · key pair · PEM · `.pem`

1. [AWS Console](https://console.aws.amazon.com/) → region **us-east-1** → open **CloudShell** (`>_` icon).
2. Paste **Block 1** and press Enter once:

```bash
export AWS_DEFAULT_REGION=us-east-1

aws ec2 create-key-pair \
  --key-name soc-lab-key \
  --query 'KeyMaterial' \
  --output text > ~/soc-lab-key.pem

chmod 400 ~/soc-lab-key.pem
ls -la ~/soc-lab-key.pem
```

3. CloudShell → **Actions** → **Download file** → enter `soc-lab-key.pem` → save to a folder on your PC (e.g. **Downloads**).

**Expected success (Block 1):**

```text
-r--------. 1 cloudshell-user cloudshell-user 1675 ... /home/cloudshell-user/soc-lab-key.pem
```

The file is ~1.6 KB. If `create-key-pair` fails with duplicate name, run **Cleanup** first.

**Or use the Console:** **EC2 → Key Pairs → Create key pair** · Name `soc-lab-key` · RSA · `.pem` · Download.

**Done when:** `ls` shows `-r--------` and the file is saved on your PC (before you close CloudShell).  
**Screenshot:** `step-01-key.png` — CloudShell after `ls -la`, or File Explorer showing downloaded `.pem` (crop any personal paths if sharing).  
**Stuck?** Wrong region → **us-east-1**. `Permission denied` writing `.pem` in CloudShell → **Cleanup** or `rm -f ~/soc-lab-key.pem`, then Block 1 again. No download menu → Console method above.

## Step 2 — Launch EC2 (Block 2)

**Story:** Act 2 — build the building; bouncer allows ports 22 and 80 only.  
**Terms:** EC2 · VPC · Security Group · soc-lab-sg

1. In the **same** CloudShell session, paste **Block 2** and press Enter once:

```bash
export AWS_DEFAULT_REGION=us-east-1

aws ec2 create-security-group \
  --group-name soc-lab-sg \
  --description "SOC Lab SG" | cat

VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' --output text)

SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=soc-lab-sg Name=vpc-id,Values=$VPC_ID \
  --query 'SecurityGroups[0].GroupId' --output text)

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID --protocol tcp --port 22 --cidr 0.0.0.0/0 | cat

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID --protocol tcp --port 80 --cidr 0.0.0.0/0 | cat

SUBNET_ID=$(aws ec2 describe-subnets \
  --filters Name=vpc-id,Values=$VPC_ID Name=default-for-az,Values=true \
  --query 'Subnets[0].SubnetId' --output text)

INSTANCE_ID=$(aws ec2 run-instances \
  --image-id ami-098e39bafa7e7303d \
  --instance-type t2.medium \
  --key-name soc-lab-key \
  --network-interfaces "AssociatePublicIpAddress=true,DeviceIndex=0,SubnetId=$SUBNET_ID,Groups=$SG_ID" \
  --query 'Instances[0].InstanceId' --output text)

echo "Instance ID: $INSTANCE_ID"
```

2. Copy **Instance ID** (e.g. `i-0abc123def456789`) to your worksheet.

**Expected success (Block 2):** JSON with `"Return": true` for ports 22 and 80, then:

```text
Instance ID: i-0abc123def456789
```

Ignore duplicate-resource errors if you are re-running without cleanup. Scroll past JSON — the line that matters is `Instance ID:`.

**If you must split Block 2** (same session only):

*Chunk A — through port 80 rules* · *Chunk B — `SUBNET_ID`, `run-instances`, `echo`*

**Or use the Console:** **EC2 → Launch Instance** · Amazon Linux 2023 · `t2.medium` · key `soc-lab-key` · allow **SSH** and **HTTP** from `0.0.0.0/0`.

**Done when:** You see `Instance ID: i-...` and EC2 console shows instance **running**.  
**Screenshot:** `step-02-instance.png` — CloudShell showing `Instance ID:` or EC2 **Instances** list (blur account ID if sharing).  
**Stuck?** `InvalidAMIID` → wrong region. `InvalidKeyPair` → Step 1 first. Pager stuck → **`q`**. No instance ID → re-paste Chunk B.

## Step 4 — VS Code SSH config & key permissions

**Story:** Act 4 — program your key fob; lock it so only you can use it.  
**Terms:** SSH config · IdentityFile · icacls · PEM permissions

### 4a — SSH config

1. Open **VS Code** → Remote Explorer → **SSH** → gear → **Open SSH Configuration File**.
2. Paste — replace `YOUR_PUBLIC_IP_HERE` and `YOUR_USERNAME`:

```ssh-config
Host SOC-Instance
  HostName YOUR_PUBLIC_IP_HERE
  IdentityFile "C:\Users\YOUR_USERNAME\.ssh\soc-lab-key.pem"
  User ec2-user
```

3. Save. Confirm **SOC-Instance** appears under SSH.

**Tip:** Right-click your `.pem` file in File Explorer → **Copy as path** — use that exact path for `IdentityFile` if you keep the key in Downloads instead of `.ssh`.

### 4b — Secure the key on Windows (required)

OpenSSH on Windows **rejects** private keys that other users or groups can read. VS Code may show:

- `UNPROTECTED PRIVATE KEY FILE!`
- `Permissions for '...soc-lab-key.pem' are too open`
- `Load key "...": bad permissions`
- `Permission denied (publickey,gssapi-keyex,gssapi-with-mic)`

Extra groups (e.g. **Users**, **Authenticated Users**, or IDE sandbox accounts) on the file cause this — not a wrong password.

**Option A — key in `.ssh` (recommended)**

```powershell
New-Item -ItemType Directory -Force -Path "$env:USERPROFILE\.ssh" | Out-Null
Move-Item "$env:USERPROFILE\Downloads\soc-lab-key.pem" "$env:USERPROFILE\.ssh\soc-lab-key.pem"

$key = "$env:USERPROFILE\.ssh\soc-lab-key.pem"
icacls $key /inheritance:r
icacls $key /grant:r "${env:USERNAME}:R"
icacls $key /remove "Users" "Authenticated Users" 2>$null
icacls $key
```

The last command should show **only your Windows username** with `(R)` — no `Users`, `Authenticated Users`, or other group entries.

**Option B — key stays in Downloads**

```powershell
$key = "$env:USERPROFILE\Downloads\soc-lab-key.pem"
icacls $key /inheritance:r
icacls $key /grant:r "${env:USERNAME}:R"
icacls $key /remove "Users" "Authenticated Users" 2>$null
icacls $key
```

Update `IdentityFile` in your SSH config to match wherever the key lives.

> **Important:** The `.pem` on your PC must be the **same** key pair used when the instance was launched. If you delete and recreate `soc-lab-key` in AWS after launch, you must **terminate the instance**, run **Cleanup**, and repeat Steps 1–2 — the running instance will not accept a new key.

**Done when:** Config has your real public IP and key path; `icacls` shows only your user can read the `.pem`.  
**Screenshot:** `step-04-vscode-config.png` — SSH config with **HostName** and **IdentityFile** filled in (redact IP/username if sharing). Do **not** screenshot the `.pem` contents.  
**Stuck?** Wrong path → **Copy as path** from File Explorer. `publickey` after ACL fix → key not downloaded or wrong instance/key pair.

## Cleanup

Use this when you finish the lab **or** when you want to wipe prior attempts and start over (see optional note above).

Run this in CloudShell to remove the lab instance, security group, key pair, and local key file. Also delete your local `soc-lab-key.pem` on your PC (Downloads or `.ssh` folder).

```bash
export AWS_DEFAULT_REGION=us-east-1

echo "=== 1. Terminate lab instances ==="
INSTANCE_IDS=$(aws ec2 describe-instances \
  --filters "Name=key-name,Values=soc-lab-key" \
            "Name=instance-state-name,Values=pending,running,stopping,stopped" \
  --query 'Reservations[].Instances[].InstanceId' \
  --output text)

if [ -n "$INSTANCE_IDS" ] && [ "$INSTANCE_IDS" != "None" ]; then
  echo "Terminating: $INSTANCE_IDS"
  aws ec2 terminate-instances --instance-ids $INSTANCE_IDS | cat
  echo "Waiting for termination (1-2 min)..."
  aws ec2 wait instance-terminated --instance-ids $INSTANCE_IDS
  echo "Instances terminated."
else
  echo "No lab instances found."
fi

echo "=== 2. Delete security group ==="
VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' --output text)

SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=soc-lab-sg Name=vpc-id,Values=$VPC_ID \
  --query 'SecurityGroups[0].GroupId' --output text 2>/dev/null)

if [ -n "$SG_ID" ] && [ "$SG_ID" != "None" ]; then
  aws ec2 delete-security-group --group-id $SG_ID | cat && echo "Security group deleted."
else
  echo "No soc-lab-sg found."
fi

echo "=== 3. Delete key pair ==="
aws ec2 delete-key-pair --key-name soc-lab-key | cat && echo "Key pair deleted." || echo "Key pair not found."

echo "=== 4. Remove local key file ==="
rm -f ~/soc-lab-key.pem && echo "Local pem removed."

echo "=== DONE — everything cleaned up ==="
```

If security group delete fails with `DependencyViolation`, wait 2 minutes and run only:

```bash
export AWS_DEFAULT_REGION=us-east-1
VPC_ID=$(aws ec2 describe-vpcs --filters Name=is-default,Values=true --query 'Vpcs[0].VpcId' --output text)
SG_ID=$(aws ec2 describe-security-groups --filters Name=group-name,Values=soc-lab-sg Name=vpc-id,Values=$VPC_ID --query 'SecurityGroups[0].GroupId' --output text)
aws ec2 delete-security-group --group-id $SG_ID | cat
```

| **Security Group** | — | Virtual firewall on your instance |

| **chmod** | Change mode | Set file permissions |
